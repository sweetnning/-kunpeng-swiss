开场 / Title

Frontend Slides · Editable

单文件 HTML 演示 + 浏览器内完整编辑运行时：拖拽、排版、重排页面、本地保存与导出。

Zero-dependency · Pitch distilled from README.md

02
叙事 / Why

生成不是定稿

  • 评审后还要改布局、改措辞、对齐品牌。
  • 只读 HTML 交付快，但迭代成本落在「重新生成」上。
  • 可编辑分支让第一版即可在浏览器里继续塑形。
03
产品 / What

可编辑的 frontend-slides 分支

  • 保留父技能的：风格探索、视口纪律、PPT 提取流程。
  • 新增：对象级拖拽/缩放、富文本、Pages 侧栏、Undo/Redo。
  • 交付仍是单文件；样式仍应按 STYLE_PRESETS 逐套实现。

04
对照 / Compare

父技能 vs 可编辑

frontend-slides

  • 更轻、更干净的只读输出
  • 适合接近定稿、体积敏感
  • 无 Pages / 无完整对象历史
  • 可选轻量文本编辑（按流程）

editable fork

  • 完整编辑运行时（默认）
  • 评审后继续改布局与内容
  • 侧栏缩略图、重排、删除
  • 保存整副 deck 结构并可导出
05
能力 / Features

核心特性（节选）

  • 零依赖：内联 CSS/JS，无需 npm。
  • Ctrl+点击多选；吸附对齐；撤销/重做。
  • 粗体/斜体/字体/字号；文本框缩放自动重排。
  • Pages：缩略图导航、重排、删除幻灯片。
  • Ctrl+S 持久化；导出剥离临时编辑态的 .html。
  • 视口安全：每页必须适配，禁止页内滚动。
06
场景 / When

何时选哪一个

选 Editable

  • 生成后仍要微调版式
  • 客户/团队反馈迭代密集
  • PPT→网页后仍要改

选父技能

  • 只读展示即可
  • 追求更小文件体积
  • 不需要编辑器界面
07
信念 / Principles

设计理念

  • 用「看见选项」建立审美，而不是抽象问卷。
  • 第一版就要能编辑；依赖越少，保质期越长。
  • 放不下就拆页，而不是藏起溢出。
  • 辨识度优先：精选预设对抗模板感。
  • 可编辑是运行时加成，不是统一成套土版式。
